## 1. 배경 및 문제 정의

### 1-1. 업로드 3,177건 끊김 문제 (크리티컬)
- **증상**: 11,228건 엑셀 업로드 시 3,177건만 INSERT, 나머지 무시됨
- **원인**: upload.ts BATCH_SIZE=4000, 레코드당 18개 파라미터 → 4000×18=72,000 > PostgreSQL 한계 65,535
- **에러 로그**: `Batch insert error: bind message has 6464 parameter formats but 0 parameters` (upload.ts:379)
- **영향**: 모든 고객사에서 동일 현상 발생 중

### 1-2. 대용량 업로드 불가 (구조적)
- 고객사 데이터 규모: 30만~80만건 (최대 100만건 이상 가능)
- 현재 `/parse`에서 allData(전체 데이터)를 JSON으로 브라우저에 전송 → 메모리 폭발
- `/save`가 동기 처리 → HTTP 타임아웃, 브라우저 닫으면 중단
- 진행률 Redis 코드 있으나 실질적 미활용

### 1-3. customers UNIQUE 키 문제
- 현재: `company_id + phone` → 같은 회사 내 브랜드별 고객 구분 불가
- 화장품 본사(시세이도, 엘오케이 등) → 브랜드 10개, 고객 중복 가입
- NARS 3만 + CPB 2만 + SH 1.5만 업로드 → 중복 제거되어 4만건만 저장 (잘못된 동작)
- 각 브랜드별 독립 마케팅이 정상 → 별도 row로 저장되어야 함

### 1-4. 수신거부 구조 문제
- 현재: company_id 기준 → 브랜드별 수신거부 구분 불가
- 변경: user_id 기준 → 각 사용자가 자기 수신거부 목록 관리
- 080 API 콜백: 해당 080번호 사용하는 모든 user에게 broadcast INSERT

### 2-2. customers UNIQUE 키 변경

#### 변경 내용
```
현재: UNIQUE (company_id, phone)  — idx_customers_company_phone
변경: UNIQUE (company_id, COALESCE(store_code, '__NONE__'), phone)
```

- store_code 있는 고객사 (시세이도 등): company_id + NARS + 01012345678 / company_id + CPB + 01012345678 → 별도 row
- store_code 없는 고객사 (아이소이 등): company_id + __NONE__ + 01012345678 → 기존과 동일 1건
- PostgreSQL UNIQUE에서 NULL은 서로 다른 값 취급 → COALESCE 필수 (안 하면 같은 번호 무한 INSERT)

#### DDL
```sql
-- 기존 인덱스 삭제
DROP INDEX IF EXISTS idx_customers_company_phone;

-- 새 UNIQUE 인덱스 생성
CREATE UNIQUE INDEX idx_customers_company_store_phone 
ON customers (company_id, COALESCE(store_code, '__NONE__'), phone);
```

#### 영향받는 모든 ON CONFLICT 구문 수정
- upload.ts: `ON CONFLICT (company_id, phone)` → `ON CONFLICT (company_id, COALESCE(store_code, '__NONE__'), phone)`
- sync.ts: 동일 변경
- customers.ts: 중복 체크 로직 변경
- campaigns.ts: 타겟 추출 시 store_code 고려

### 2-3. 수신거부 user_id 기준 전환

#### 현재 구조
```
unsubscribes: company_id + phone → 회사 전체 수신거부
opt_outs: company_id + phone → 회사 전체 수신거부
```

#### 변경 구조
```
unsubscribes: user_id + phone → 사용자별 수신거부
opt_outs: user_id + phone → 사용자별 수신거부
```

#### DDL
```sql
-- unsubscribes에 user_id 추가
ALTER TABLE unsubscribes ADD COLUMN user_id uuid REFERENCES users(id);

-- opt_outs에 user_id 추가
ALTER TABLE opt_outs ADD COLUMN user_id uuid REFERENCES users(id);

-- 인덱스 변경 (기존 company_id 기반 → user_id 기반)
-- 기존 인덱스 확인 후 교체 필요
```

#### 발송 시 수신거부 체크 로직 변경
- 현재: 회사(company_id) 수신거부 목록에서 제외
- 변경: 캠페인 생성 사용자(user_id)의 수신거부 목록에서 제외

#### 080 API 콜백 변경
- 080 수신거부 접수 시: 해당 080번호를 사용하는 고객사의 **모든 user**에게 INSERT
- 조회: callback_numbers 테이블에서 해당 080번호 → company_id → users → 전원에게 insert

### 2-4. 요금제 기준
- max_customers = 관리 중인 DB 건수 기준 (브랜드별 중복 포함)
- NARS 3만 + CPB 2만 + SH 1.5만 = 6.5만건이 기준
- 플랜 초과 체크 로직 변경 없음 (customers 테이블 COUNT 그대로)

## 3. 기타 에러 수정 (에러 로그에서 발견)

### 3-1. 스팸필터 spam_check_number NOT NULL 에러
- **파일**: spam-filter.ts:75
- **에러**: `null value in column "spam_check_number" violates not-null constraint`
- **원인**: 080 치환 로직 제거 후 spam_check_number에 null 전달
- **수정**: DB 컬럼 NOT NULL 제거 또는 빈 문자열 기본값

### 3-2. 발송결과 타임스탬프 파싱 에러
- **파일**: results.ts:121, 212
- **에러**: `invalid input syntax for type timestamp: "2026-02-20 00:00:00 00:00:00"`
- **원인**: 타임존 `+00:00`이 `00:00:00`으로 들어감 (+ 기호 누락)
- **수정**: 날짜 파라미터 생성 로직에서 타임존 포맷 수정

### 3-3. SMS 상세조회 MySQL 에러
- **파일**: admin.ts:1081
- **에러**: `Incorrect arguments to mysqld_stmt_execute` (LIMIT ? OFFSET ?)
- **원인**: MySQL prepared statement에 LIMIT/OFFSET이 문자열로 전달됨
- **수정**: `Number(limit)`, `Number(offset)` 캐스팅
