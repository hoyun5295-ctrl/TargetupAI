1. 개요

    본 문서는 문자 에이전트를 이용하여 문자를 전송하려는 사용자를 위한 설명서이다.
    본 문자 에이전트는 SMS(단문), LMS(장문), MMS(첨부파일포함), KMS(알림톡),FMS(친
   구톡),GMS(친구톡 파일첨부)을 지원하며 실행파일로 동작하는 Agent이다.

2. QTmsg Agent 프로그램 설치환경

     1) 사전설치
           본 Agent는 실행파일로 동작하는 버전으로서 사전패키지 설치는 없으며 운영체제
           에 맞는 실행파일을 공급받아야 한다.

     2) DBMS지원
           Oracle(9이상), MSSQL(2000이상), Mysql(maria,오로라포함,mysql5.8이상권장),
           mongo(3.4이상)

     3) 실행가능 운영체제(인텔,ARM7:라즈)
           리눅스 CENT7.5이상(Redhat/Fedora포함), Ubuntu 18.04이상

3. QTmsg Agent 설치 및 실행/종료 방법

     -. 설치
          운영실에서 배포된 zip파일을 운영대상 서버에 전송한 뒤에 압축을 풀면 된다.

     -. Agent 프로그램 실행
             Agent 패키지 파일을 풀면 해당 폴더(directory)에 아래 파일이 있는데 이를
             실행하면 된다.
            리눅스: ./start.sh
            윈도우 : command창에서 1회만 실행하여 테스트할 때는 test.cmd를 클릭하여
                         테스트 한다.
                  윈도우 서비스로 설치 하려면 install_winservice.bat을 관리자 권한으로
                  더블클릭한다.
             *. 실행결과는 logs폴더(directory) 밑에 …txt파일 내용을 확인한다.

     -. Agent 프로그램 종료
            bin폴더(directory)에서 다음을 실행한다.
            리눅스 : ./stop.sh
            윈도우 : command창에서 실행했다면 command창을 닫는다.
                     윈도우 서비스에서 설치를 해제하려면 uninstall_winservice.bat를
                     더블클릭한다.

-. 리눅스 systemctl 기능관련 서비스 등록 방법
    본 패키지 파일 배포시 함께 배포되는 doc밑에qtmsg.service파일내용에서 아래의
    exec stop start 내용을 설치된 directory path로 변경하여 저장한 뒤에
    /etc/systemd/system directory로 copy한뒤에 동작 명령을 수행한다.
    (qtmsg 파일은 Agent실행파일임)
    ---- exec path관련 ---
    ExecStart=/home/qtmsg/4.0.x.x/qtmsg start
    ExecStop=/home/qtmsg/4.0.x.x/qtmsg stop
    ---- 동작명령 ----
    위 copy명령 포함하여 명령은 전부 리눅스 root 사용자로 로그인하여 수행한다.
    systemctl start qtmsg[엔터] : 실행
    systemctl stop qtmsg[엔터] : 중지
    systemctl enable qtmsg[엔터] : 리눅스 재기동시 자동실행
    systemctl disable qtmsg[엔터] : 리눅스 재기동시 자동실행 중지

4. QTmsg 동작위한 기본설정

  -. 개요
       본 Agent는 중계서버에 접속하기 위한 Bind ID, Bind 비밀번호와 DBMS접속
     정보가 사전에 세팅되어 있어야 동작한다. 세팅파일은 conf밑에 qtmsg.xml이다.
         Bind ID와 Bind 비밀번호는 사전에 운영실에서 해당 ID를 발급한 뒤에 패키지파일
         에 사전에 세팅하여 배포하므로 고객사에서 세팅하지 않아도 되며 DBMS만 세팅하여
         운영하면 된다.
         *. Agent 중복실행제한 : 중계서버에서는 bind id가 유일해야 하기 때문에
             동일 Bind ID로 이중실행(다른서버포함)되는 경우 문제가 되므로 한 개만
             실행하며 강제로 실행하는 경우 실행이 안된다.
               필요에 따라서 2개 이상 Agent실행이 필요한 경우 운영실에 요청하여 추가
               발급받고 운영실에 세팅 지원요청한다.
               (2개실행:관리 포트를 별도로 세팅 : qtmsg.xml파일 admin_port tag숫자)
         *. Agent 2개를 이용한 이중화 관련 설명
             Agent를 상시 2개로 운영하여 나누어서 발송되도록 하며 1개가 다운되면 다른
               1개가 자동으로 발송되도록 이중화 구현이 가능하며 이렇게 하려는 경우
               rsv1 상태 값에 대하여 양쪽 Agent가 충돌하지 않도록 커스톰이 필요하다.
               본 사항이 필요한 경우 운영실에 별도 요청하여 xml파일의 상태
               값 커스톰된 2개의 Agent를 받도록 한다.

    -. 발송요청 및 결과 테이블(DBMS)생성
         발송요청 테이블(SMSQ_SEND)과 결과 테이블(SMSQ_SEND_YYYYMM)은
         2개 모두 Agent처음 실행 시 DB에 없으면 자동으로 생성된다.
         ( table 생성시 option이 필요한 경우 qtmsg.xml의 create sql을 수정한다 )
         단, mongo DB의 경우 nosql이므로 따로 생성하지 않는다.

  -. DBMS관련 세팅은 DBMS ip주소, DBMS유형, DB name(service name), DB user,
     DB User password, port번호(DBMS socket listen port)를 conf밑에 qtmsg.xml
     파일에 세팅한 뒤에 저장 하는 것이다.
     *. 테이블명은 필요에 따라서 변경하여 사용할 수 있다.(2개 Agent실행시 각각 별도
     테이블 운영가능하며 이중화인 경우 동일테이블이 된다.)
     -------------- DBMS설정 관련 : 파일 : conf 하부에 qtmsg.xml---------
     <DB>
     <!-- DB서버의 IP주소 또는 HOST명 넣음 -->
     <server>127.0.0.1</server>
     <!-- DB종류 mssql, mssql2000,mysql, oracle,mongo 중 한개 지정 -->
     <type>mysql</type>

5. QTmsg 문자 발송요청 방법

   -. 개요
      문자 발송요청은 SMSQ_SEND테이블에 문자 발송내용 데이터를 insert시키면 되며
      Sql은 내용은 DBMS별로 아래 예시 참조한다.
     전송결과는 일정시간(SMS,LMS:최장2일)이내 이통사로부터 수신된 결과가
     SMSQ_SEND_YYYYMM (월별)테이블에 저장된다.

  -. 문자 전송 상태 확인
       전송상태는 Rsv1필드로 확인할 수 있으며 다음과 같은 상태를 가진다.
         1: 발송대기, 2:Agent처리중, 3:서버로전송요청완료,4:결과수신완료,5:월별처리완료

   -. 발송요청 DBMS테이블 명세(그림1)

   사용목적) 본 Agent를 이용하여 문자발송시 문자별로 발송하는 중계사가 다른경우 식

-. 카카오톡 발송 실패 결과 후 일반문자 전환재발송 관련
   설명) 카카오톡 발송 후 실패 시 일반문자로 발송요청하려면 k_next_type field에
   S,L,A,B(세부사항 테이블설명서 참조:그림1)의 값을 넣으면 된다.
   방법2가지) 일반문자 전환 재발송은 Agent주관하여 진행하는 방법과 서버에서 진행 한
   뒤 결과만 내려주는 방법이 있으며 default는 Agent주관 방법이다. 서버에서 재발송
   진행한 뒤 결과만 내려주는 경우에는 결과코드는 일반문자 성공의 경우 SMS는
   7830,LMS는 7831이며 Agent주관의 경우에는 카톡과 다른 DB record로서 6(SMS)
   또는 1000(LMS/MMS)로 입력된다. 해당 record에 원래 카톡문자 순번값은 k_oriseq
   에 저장된다.
   결과 확인 및 재발송 절차설명) Agent주관 방법
   1) 카카오톡 발송 실패는 3초 이내에 Agent에서 중계서버로부터 결과를 수신하여 DB
        의 status_code필드에 결과 값을 세팅한다.
   2) Agent는 결과 세팅 후 바로 일반 문자로 발송하라는 지침이 있는 경우 일반문자로
        발송요청하고 해당 근거를 DB에(SMSQ_SEND)에 insert하여 준다. 이때 문자유형
        은 발송지침(k_next_type)에 근거하여 msg_type에 해당 값을 자동 세팅한다.
   3) 이후 Agent는 일정 시간 뒤 일반문자 발송 결과를 수신하면 DB의 status_code에
        결과 코드 값을 세팅한다.

-. 친구톡 파일첨부 발송방법(G:msg_type)
   SMSQ_SEND테이블의 File_name1필드에 파일 name(path포함)을 넣어서 insert하
   면 된다. 친구톡 파일 첨부의 경우에는 아래의 사양기준이다(카카오톡사 제한)
       권장 사이즈 : 720px*720px (일반)
       제한 사이즈 - 가로 500px 미만 또는 가로:세로 비율이 2:1 미만 또는 3:4 초과시
   업로드
       파일형식 및 크기 : jpg, png / 최대 500KB
   *. 주의 : 파일 첨부기능을 이용하려는 경우에는 사전에 운영자에게 문의하여 해당 기
   능 중계에서 지원되는지 사전 확인한다.(사유:중계서버 사유에 따라서 기능은 막힐 수
   있다)

-. 카카오톡 sender key관련(발송프로필키)
   카카오톡 sender key는 중계업체가 변경되는 경우 카카오회사로부터 새로 발급받게 되
   며 해당 sender key는 다음의 3가지 방법으로 문자발송시 세팅하여 카톡문자 전송한
   다.
    1) 중계사 서버에 템플릿 코드 매핑관리를 통하여 템플릿코드별로 sender key를 등
          록하여 알아서 발송함. 이 경우 agent side(고객사 DB등)에서는 신경쓰지 않아도
          되며 템플릿 코드를 참조하는 알림톡(KMS)만 해당하고 친구톡은 아래 참조한다.
    2) Agent가 바라보는 문자 DB테이블에 다양한 문자별로 여러 개의 sender key로서
          문자를 발송하는 친구톡등의 경우에는 새로운 중계사로 변경하는 경우 신규 발급
          된 sender key를 고객사 발송프로그램에서 DB에 발송 데이터 입력하여 발송요청
          할 때 새것으로 해야 한다.
    3) 고객사에서는 새로운 sender key로 발송프로그램을 변경하는 작업이 어려운 경우
          에는 xml커스토마이징 처리를 통하여 새로운 sender key로 select(fetch) sql수
          준에서 변경이 가능하므로 본 중계서버 운영실에 요청하면 세팅(커스톰)가능하다.
          이 경우 sender가 새롭게 추가되거나 변경되거나 하는 경우에 매번 xml파일을
          수정하고 agent를 재기동 해야 한다. 변경이 없는 경우에 추천한다.

6. MMS문자 첨부파일 추가정보

        1) 문자의 MMS 첨부파일 크기는 300 kbyte를 권장한다.(이통사 권장사항임)
                이통사 권장은 300 kbyte이나 실제로 스마트폰으로 전송하면 1mega byte도
                잘 수신되나 피처폰 고려하여 300kbyte까지 보장한다.

         2) 문자의 MMS 첨부파일 개수의 경우 Agent는 5개까지 가능하며 이통사 보장 최
         소 건수는 3건이며 이통사에 따라서 5개까지 가능할 수 도 있다.
        3) MMS첨부파일 확장자 및 유형

           이통사에서는 jpg를 표준지원 형식으로 보장하며 png확장자 파일은 중계(또는 통
            신사)에서 거절되는 경우가 있으므로 jpg로만 보낼 것을 권장한다.
           단, GIF형식은 지원하지 않는다.(이통사 제약임)
         4) 웹사이트보관 첨부파일 추가하여 문자발송 하는 기능
           DB table의 첨부파일 field에 아래와 같은 형태로 넣으면 해당 웹사이트에서 해당
           파일을 다운받아서 보내준다. 단, 웹의 특성상 문자 건당 웹통신(socket open)이
           발생하므로 속도는 1건당 최소 0.1초이상 걸리므로 대량발송 시 속도가 많이 저하
           될 수 있다.

                예) http://www.naver.com/abcd/kkk.jpg
        5) 문자유형을 ‘L’로 한 뒤에 파일을 첨부(file_name#에 파일명 추가)하여도 이는

             무시된다.
         6) 문자유형을 ‘M’으로 한 뒤에 파일을 첨부하지 않거나 file_name#필드의 파일명
         을 이용하여 파일이 실제 있는지 점검한 뒤에 파일이 없는 경우에는 에러 처리한
         다.(에러 코드 : 9007). 단, 문자유형을 ‘M’으로 하고 LMS와 MMS가 파일 첨부 여
         부에 따라서 자동으로 동작하게 하는 방법이 있는데 아래 option을 yes로 하면 된
         다.(파일: qtmsg.xml). 수정 후 Agent재기동 해야 반영됨.(아래동일)

               <AutoSelect_Mms_Lms_Flag>no</AutoSelect_Mms_Lms_Flag>
         7) 첨부파일의 크기는 최대 2메가 바이트로 제한하며 xml파일에서 수정할수 있으며
         아래의 tag를 수정하면 된다. 4메가 이상은 통신상 중계서버에서 안된다.
