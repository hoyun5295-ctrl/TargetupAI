## 🔴 A. 매뉴얼 수정 (비토 수정)

### A-1. 슈퍼관리자 도메인 노출 삭제

| 위치 | 현재 내용 | 문제 |
|------|----------|------|
| 6.3절 (153행) | `sys.hanjullo.com에서 Agent 상태가 녹색(정상)으로 표시` | 슈퍼관리자 URL이 고객사에 노출됨 |

- **수정:** `sys.hanjullo.com에서~` → `고객사 관리자 페이지에서 Agent 상태가 녹색(정상)으로 표시` 로 변경

## 🟡 B. Agent측 보안 수정 요청 (비토 수정)

### B-1. 웹 UI 마법사 바인딩 주소 확인 (중요)

- Windows 설치 시 `http://localhost:9876` 자동 오픈
- **확인 필요:** 실제로 `127.0.0.1:9876`에만 바인딩되는지?
- `0.0.0.0:9876`으로 바인딩되면 같은 네트워크 내 누구나 접근 가능
  → DB 접속 정보, API 키 설정 화면이 외부에 노출됨
- **요청:** `127.0.0.1`만 바인딩 확인, 아니면 즉시 수정

### B-2. .env 프로덕션 사용 경고

- 현재 .env(평문)와 config.enc(암호화) 둘 다 지원
- 고객사 담당자가 "개발용" .env를 운영에 그대로 쓸 가능성 있음
- **요청:**
  - Agent 시작 시 `.env` 파일 감지 + 프로덕션 모드면 경고 로그 출력
  - 매뉴얼 .env 섹션에 "⚠️ 운영 환경에서는 반드시 설치 마법사(config.enc) 사용" 경고 강화

### B-3. 자동 업데이트 download_url 검증

- 현재 서버 응답의 download_url을 그대로 다운로드
- 만약 DB가 탈취되면 악성 바이너리 URL 삽입 → 전 고객사 Agent 감염 가능
- **요청:**
  - Agent에서 download_url 도메인 화이트리스트 검증 (우리 도메인만 허용)
  - checksum 검증을 선택이 아닌 **필수**로 변경 ("서버 제공 시" 조건 삭제)

### B-4. 배치 사이즈 문서 불일치

- 매뉴얼: 4,000건씩 배치 (6.1절)
- 서버 제한: 5,000건
- **요청:** Agent 실제 배치 사이즈가 4,000이면 OK. 서버 제한(5,000)과 불일치만 확인

## 🔵 C. 서버측 수정사항 (우리가 수정)

### C-1. Sync API rate limit 추가 (우선)

- 현재 `/api/sync/*`에 rate limit 없음
- API Key/Secret 브루트포스 대입 공격 방어 필요
- **구현:** Nginx `limit_req` 또는 Express rate-limit 미들웨어
  - 인증 실패: IP당 분당 10회 초과 시 차단
  - 인증 성공: 회사당 분당 60회 제한

### C-2. 동시 동기화 제한

- 여러 고객사가 동시에 full sync 시 서버 부하
- **구현:** 고객사별 동시 동기화 1개 제한 (진행 중이면 429 응답)

### C-3. sync_releases 운영 프로세스

- 테이블 생성 완료. 비토가 v1.3.0 최종 빌드 완료 시 릴리스 레코드 등록 필요
- 슈퍼관리자 UI 또는 수동 SQL INSERT로 관리
